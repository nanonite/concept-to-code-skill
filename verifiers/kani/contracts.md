# Kani Contract Patterns

Use Kani for bounded reachability, panic-freedom, parser roundtrips, and
unsafe or FFI boundary checks. Generated stubs use `cfg_attr` so contracts
disappear outside Kani builds.

## Function Contracts

`emit_stubs.py` emits preconditions from `constraints[].logic`:

```rust
#[cfg_attr(kani, kani::requires(input.len() <= MAX_LEN))]
#[cfg_attr(kani, kani::ensures(|ret| ret.is_ok()))]
pub fn parse(input: &[u8]) -> Result<Model> { unimplemented!() }
```

Good Kani logic is side-effect-free Rust over bounded values:

- slice/index bounds: `i < xs.len()`, `xs.len() <= 32`;
- finite floats: `x.is_finite()`, `x >= 0.0`;
- parser shape: `!tokens.is_empty()`, `depth <= MAX_DEPTH`;
- roundtrip postconditions: `parse(emit(ret)).is_ok()` for small bounded
  values.
- nonlinear arithmetic preconditions: bound operands before multiplication,
  e.g. `a <= MAX`, `b <= MAX`, `a * b <= LIMIT`, and product-plus-offset
  relations such as `a * b + c <= LIMIT`; keep multiplication over bounded
  integer domains so the solver has finite, explicit ranges.

## Proof Harnesses

Keep one property per harness. Construct symbolic inputs with `kani::any()`,
then bound them with `kani::assume()` before calling production code.

```rust
#[cfg(kani)]
mod proofs {
    use super::*;

    #[kani::proof]
    #[kani::unwind(8)]
    fn parser_never_indexes_past_token_buffer() {
        let len: usize = kani::any();
        kani::assume(len <= 8);
        let token: u8 = kani::any();
        let input = vec![token; len];
        let _ = parse_tokens(&input);
    }
}
```

Before writing or changing a harness, state the one bounded property it
proves: panic freedom, reachability, roundtrip behavior, contract
preservation, or unsafe/FFI boundary behavior. If a harness would otherwise
cover multiple obligations, split them into separate harnesses instead of
combining assertions into one broad proof.

Mirror the production API contract exactly. Use `kani::assume` only for
caller obligations or input-domain restrictions that are genuinely part of
the intended contract. Never add an assumption just to remove a failing path
that production code is supposed to handle. Check every assumption for
vacuity: after all assumptions, the harness must still admit meaningful
inputs, reach the production call, and exercise the code path tied to the
stated property.

Make every symbolic input finite before invoking production code. Bound
vector and slice lengths, loop counts, recursion depth, enum discriminants,
numeric ranges, parser depth, allocation sizes, and symbolic indexes used for
access. Use small explicit bounds that still exercise edge cases such as
empty, singleton, maximum-length, and boundary-index inputs when relevant.
For symbolic vectors, arrays, strings, and token streams, construct bounded
collections deliberately: choose a symbolic length within a small maximum,
fill elements from `kani::any()`, and assume only required element-level
invariants before the call.

Use `assert!` only for obligations the implementation must satisfy under the
harness assumptions. For pure panic-freedom checks, call the production
function and avoid semantic assertions unless required. Keep Kani-only code
behind `#[cfg(kani)]` or `cfg_attr(kani, ...)` so normal builds and runtime
behavior remain unchanged.

For parser and serializer tasks, prefer small bounded byte or token inputs
and check panic freedom, rejection without panic, or roundtrip acceptance for
values generated by the serializer. Do not require parse success for
arbitrary symbolic input unless the parser explicitly promises total
acceptance.

### Dispatch and Equivalence Harnesses

For dynamic dispatch, verify the call site that uses the erased type, not
only the concrete implementation. Cover `&dyn Trait` and `Box<dyn Trait>`
paths so vtable resolution agrees with direct calls to the concrete impl
under the same bounded inputs. If drop or auto-trait behavior matters,
include the owned `Box<dyn Trait>` case that exercises destruction.

For closure dispatch, add harnesses for captured environments through `dyn
Fn`, `dyn FnMut`, and `dyn FnOnce`. Bound captured state and call arguments
separately; for `FnMut`, check state changes across the bounded call
sequence, and for `FnOnce`, verify the single consuming call path.

For safe/unsafe boundaries, use faithful emulation: build a small safe model,
call the unsafe implementation only on in-bounds inputs, and assert that both
results and observable state agree. For raw pointer APIs, make boundary
preconditions explicit and minimal: pointer non-nullness, alignment,
initialized memory, slice/index bounds, length validity, and aliasing or
non-overlap requirements before entering the unsafe block. Assert only the
boundary behavior that must hold after those preconditions are met.

## Limits

Kani does not handle unbounded recursion, hard quantified proofs, or
algebraic invariants over generic tree structures. Use Creusot for structural
math and Verus for loop-heavy numerical kernels. If CBMC reports unwind
exhaustion, first reduce the bound, add or correct the unwind annotation, or
isolate the specific loop or recursion being verified. If the property
requires structural induction, unbounded recursion, quantified reasoning, or
large algebraic invariants, choose a smaller bounded obligation or move it to
a verifier better suited to that proof.

## Reporting Results

When a Kani proof passes, report its scope precisely: it holds only under the
harness assumptions, explicit bounds, and unwind limits. Do not present a
bounded proof as an unbounded guarantee. Treat every failing proof as a
signal to inspect: prefer fixing production code, correcting an overbroad
assertion, narrowing the harness to the requested property, or making a real
precondition explicit before weakening the harness assumptions. When
possible, run or reference the exact Kani harness affected by a change. If
you cannot run it, make that limitation explicit and still keep the harness
internally bounded, non-vacuous, and aligned with the requested property.

Prefer the smallest local adjustment that preserves an existing proof style
and satisfies the requested property; do not redesign stable proof patterns
for abstraction or generality alone.
